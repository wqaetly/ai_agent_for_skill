# 技术分析框架提示词

## 分析方法论

### 系统性分析流程
1. **数据结构审视** → 识别核心数据和关系
2. **数据流追踪** → 分析流向、所有权和修改权限
3. **效率审查** → 找出冗余操作和性能瓶颈
4. **架构决策权衡** → 平衡各种非功能需求

## 数据结构审视框架

### 核心数据识别
**分析要点**:
- 什么是系统的核心数据实体？
- 数据间的关系类型和依赖方向？
- 数据的生命周期管理策略？
- 数据一致性和完整性保证机制？

**关键问题**:
- 核心数据是否被恰当抽象？
- 数据冗余是否合理？
- 数据访问模式是否高效？
- 数据结构是否支持未来扩展？

### 数据流追踪
**分析维度**:
- **数据流向**: 输入→处理→输出的完整路径
- **所有权**: 谁创建、修改、删除数据？
- **权限控制**: 谁可以访问什么数据？
- **变更传播**: 数据变更如何影响其他组件？

**检查清单**:
- [ ] 数据流是否清晰可追踪？
- [ ] 数据变更是否有适当的审计机制？
- [ ] 并发访问是否得到妥善处理？
- [ ] 数据泄露风险是否得到控制？

### 效率审查
**审查重点**:
- **时间复杂度**: 算法和数据操作的效率
- **空间复杂度**: 内存使用和存储效率
- **网络效率**: 数据传输和通信开销
- **I/O效率**: 磁盘读写和数据库操作

**优化策略识别**:
- 索引优化机会
- 缓存策略改进
- 批处理优化
- 懒加载实现

## 架构决策权衡分析

### 功能需求 vs 非功能需求
**性能**:
- 响应时间要求
- 吞吐量目标
- 资源使用限制
- 扩展性需求

**可维护性**:
- 代码可读性
- 模块化程度
- 测试覆盖率
- 文档完整性

**可扩展性**:
- 水平扩展能力
- 垂直扩展空间
- 新功能集成难度
- 技术栈灵活性

**安全性**:
- 认证授权机制
- 数据加密保护
- 漏洞防护措施
- 安全审计能力

### 开发效率 vs 代码质量
**开发效率因素**:
- 团队熟悉度
- 开发工具支持
- 学习曲线
- 社区生态

**代码质量要求**:
- 代码规范遵循
- 设计模式应用
- 错误处理机制
- 性能优化程度

**权衡策略**:
- MVP阶段优先开发效率
- 成熟产品优先代码质量
- 技术债务管理计划
- 重构时机选择

### 必要复杂度 vs 过度设计
**必要复杂度指标**:
- 业务需求复杂度
- 技术挑战难度
- 集成系统复杂度
- 团队协作复杂度

**过度设计警示信号**:
- 过度抽象
- 过早优化
- 不必要的灵活性
- 过度工程化

**判断标准**:
- YAGNI原则应用
- KISS原则遵循
- 最小可行实现
- 渐进式复杂化

## 质量检查标准

### 分析完整性检查
- [ ] 是否覆盖了所有核心数据结构？
- [ ] 是否分析了关键数据流？
- [ ] 是否识别了性能瓶颈？
- [ ] 是否评估了架构权衡？
- [ ] 是否提供了具体改进建议？

### 分析质量检查
- [ ] 分析是否有充分的数据支撑？
- [ ] 结论是否逻辑严密？
- [ ] 建议是否可执行？
- [ ] 风险评估是否全面？
- [ ] 优先级判断是否合理？

- 技术方案选型和对比
- 性能优化和瓶颈分析
- 系统设计和重构规划
- 技术债务评估和管理
